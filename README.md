# 🧪 Generate Test Cases — AI 驱动的测试用例生成器

> 一个 [Kiro](https://kiro.dev) Skill，从需求文档自动生成专业测试用例（XMind 格式），具备深度解析、质量自检、自主学习能力。

## ✨ 核心特性

| 特性 | 说明 |
|------|------|
| **10 种设计方法叠加** | 核心 5 种（EP/BVA/PV/ST/EG）+ 补充 5 种（RV/BS/RL/DB/REDIS），系统性覆盖需求 |
| **质量内建** | 21 项质量自检指标 + 熔断机制，生成前拦截缺陷 |
| **需求追溯** | 自动建立需求↔用例双向追溯矩阵，覆盖率 ≥ 95% |
| **自主学习** | 持久化记忆 + 历史趋势分析 + 歧义决策复用 + Bug 反向学习，越用越准 |
| **多格式输入** | 支持 Markdown、TXT、DOCX、XLSX、PPTX、PDF、图片等多种需求文档格式 |
| **XMind 输出** | 生成符合 XMind 导入规范的测试用例文件，可直接导入 XMind 使用 |

## 📦 安装

### 前置条件

- [Kiro IDE](https://kiro.dev) 已安装
- Python 3.8+
- 可选依赖（用于 Office/PDF 文档解析）：
  ```bash
  pip install python-docx openpyxl python-pptx pdfplumber
  ```

### 安装 Skill

1. 克隆本仓库到 Kiro Skills 目录：
   ```bash
   git clone <repo-url> ~/.kiro/skills/generate-test-cases
   ```
2. 在 Kiro 中即可使用

## 🚀 快速开始

1. 准备需求文档（Markdown、Word、PDF 等格式均可）
2. 在 Kiro 聊天窗口中输入：
   ```
   帮我生成测试用例
   ```
3. Skill 会自动：
   - 询问需求文档目录和输出目录（首次使用时，后续自动记忆）
   - 读取并解析需求文档
   - 按 10 种设计方法系统性生成用例
   - 执行 21 项质量自检
   - 输出质量报告 + XMind 文件

## 🏗️ 工作流程

```
Phase 0: 项目初始化（首次运行）
    ↓
Phase 1: 读取需求（多格式支持）
    ↓
Phase 2: 解析需求（10 种设计方法叠加 + 去重）
    ↓
Phase 2.5: 解析确认（有问题时询问）
    ↓
Phase 2.6: 歧义处理（仅关键歧义）
    ↓
Phase 2.8: 质量预审（21 项自检 + 质量报告）
    ↓
Phase 3: 生成文档（JSON → XMind）
    ↓
Phase 4: 更新记忆（自学习闭环）
    ↓
Phase 5: 用户反馈学习（Bug 反向学习）
```

## 🧠 10 种测试设计方法

### 核心 5 种（按序叠加）

| 方法 | 标记 | 核心动作 |
|------|------|---------|
| 等价类划分 | `EP` | 划分有效/无效等价类 |
| 边界值分析 | `BVA` | 测试上点、离点、内点 |
| 参数验证 | `PV` | 验证必填性、类型、长度、范围 |
| 场景法 | `ST` | 基本流→备选流→异常流 |
| 错误推测 | `EG` | 特殊字符、极端值、并发 |

### 补充 5 种（深度覆盖）

| 方法 | 标记 | 核心动作 |
|------|------|---------|
| 出参验证 | `RV` | 逐字段验证响应体结构 |
| 业务状态穷举 | `BS` | 每个状态枚举值独立用例 |
| 关联关系验证 | `RL` | 凭证/资源归属匹配验证 |
| 数据库验证 | `DB` | DDL 结构 + DML 数据验证 |
| Redis 缓存验证 | `REDIS` | 缓存生成/值/TTL/更新/删除 |

## 📊 质量自检（21 项指标）

生成前自动检查，不通过则自动修正（最多 3 次）：

- 需求覆盖率 ≥ 95%
- 参数验证覆盖率
- 优先级分布（P0: 10-15%, P1: 30-40%, P2: 20-40%, P3: 10-20%）
- 设计方法叠加完整性（10 种方法逐一评估）
- 出参/状态/关联/DB/Redis 覆盖率
- 负面用例占比 ≥ 30%
- 需求分支覆盖率 ≥ 90%
- 模块均衡性、方法多样性、无冗余重复、无需求外编造
- ...

质量报告同时输出到聊天窗口和文件。

## 🔄 自主学习

### 记忆系统

```
.memory/
├── project-context.json      # 项目上下文（路径、名称）
├── terminology.json          # 领域术语库
├── generation-history.json   # 生成历史（质量趋势）
├── user-preferences.json     # 用户偏好
└── ambiguity-decisions.json  # 歧义决策记录
```

### Bug 反向学习

当测试中发现了用例未覆盖的 bug 时，告诉 Skill：

```
为什么没有生成 xxx 的用例
```

Skill 会自动：
1. 分析 bug 根因
2. 从 10 种设计方法角度回溯遗漏原因
3. 生成补充用例
4. 提出改进建议（如补充业务规则）
5. 持久化学习，避免下次再遗漏

## 🎯 支持的输入格式

| 类别 | 格式 | 读取方式 |
|------|------|---------|
| 文本文件 | `.md` `.txt` `.rst` `.html` `.htm` | 直接读取 |
| Office 文档 | `.docx` `.xlsx` `.pptx` | Python 脚本提取文本 |
| PDF 文档 | `.pdf` | Python 脚本提取文本 |
| 图片文件 | `.png` `.jpg` `.jpeg` `.gif` `.webp` `.bmp` `.svg` | 多模态输入 |

## 📝 输出格式

### XMind 节点结构

```
根节点（项目名称）
  └── 模块
        └── tc-p0: 用例标题
              ├── pc: 前置条件（可选）
              ├── 1. 操作步骤
              │     └── 1. 预期结果
              └── 2. 操作步骤
                    └── 2. 预期结果
```

### 用例 JSON 字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `模块` | ✅ | 数组，最多 8 层 |
| `用例标题` | ✅ | 动作 + 对象 + 条件/场景 |
| `优先级` | ✅ | P0/P1/P2/P3 |
| `需求ID` | ✅ | 关联的需求标识 |
| `设计方法` | ✅ | 10 种方法之一或多个 |
| `前置条件` | ❌ | 特定环境准备 |
| `步骤` | ✅ | 操作 + 预期结果数组 |
