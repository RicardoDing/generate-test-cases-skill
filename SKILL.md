# 根据需求生成测试用例

从需求文档自动生成专业测试用例，具备深度解析、质量自检、自主学习能力。

## 核心能力

- **深度解析**：多种测试设计方法（EP/BVA/PV/ST/EG + RV/BS/RL/DB/REDIS）系统性覆盖需求
- **质量内建**：覆盖率验证 + 优先级分布检查 + 自检门禁，生成前拦截缺陷
- **需求追溯**：自动建立需求↔用例双向追溯矩阵，量化覆盖率
- **自主学习**：持久化记忆 + 历史趋势分析 + 歧义决策复用，越用越准
- **交互式确认**：快速模式，关键节点人工把关

## 质量标准

生成的测试用例必须满足以下 8 维度（Phase 2.8 质量预审时逐项检查）：

| 维度 | 标准 | 检查方式 |
|------|------|----------|
| **需求覆盖** | 每条需求至少关联 1 条用例，覆盖率 ≥ 95% | 追溯矩阵计算 |
| **方法覆盖** | 每条需求至少使用 1 种设计方法，复杂需求 ≥ 2 种 | 方法分布统计 |
| **方法叠加完整性** | 每条需求按 EP→BVA→PV→ST→EG→RV→BS→RL→DB→REDIS 逐一评估，符合触发条件的方法均已应用 | 10 种方法逐项检查 |
| **优先级分布** | P0: 10-15%, P1: 30-40%, P2: 30-40%, P3: 10-20% | 分布比例检查 |
| **步骤可执行** | 每条用例的步骤明确、可操作，预期结果可验证 | AI 自检 |
| **无需求外编造** | 所有用例来源于需求文档，不凭空编造场景 | 追溯关系验证 |
| **术语一致** | 用例中使用的术语与需求文档、`terminology.json` 一致 | 术语对照 |
| **无冗余重复** | 不同设计方法产生的用例无语义重复；已覆盖的验证目标不得以不同数据状态为由重复生成 | 去重扫描 |

## 测试设计方法

> 详细定义与示例见 [TEST-DESIGN-METHODS.md](references/TEST-DESIGN-METHODS.md)
> 参数验证方法详见 [PARAMETER-VALIDATION.md](references/PARAMETER-VALIDATION.md)
> 接口深度覆盖补充方法详见 [API-DEEP-COVERAGE.md](references/API-DEEP-COVERAGE.md)

**核心 5 种方法按序叠加，再由 5 种补充方法深度覆盖**：

```
核心方法：EP 等价类划分 → BVA 边界值分析 → PV 参数验证 → ST 场景法 → EG 错误推测
补充方法：RV 出参验证 → BS 业务状态穷举 → RL 关联关系验证 → DB 数据库验证 → REDIS 缓存验证
```

| 方法 | 标记 | 核心动作 | 触发条件 |
|------|------|---------|---------|
| **EP 等价类** | `EP` | 划分有效/无效等价类，无效类单独覆盖 | 有输入范围、格式、枚举约束 |
| **BVA 边界值** | `BVA` | 测试上点、离点、内点 | 有数值/长度/时间边界 |
| **PV 参数验证** | `PV` | 验证必填性、数据类型、长度边界、取值范围 | 有输入参数的接口 |
| **ST 场景法** | `ST` | 基本流→备选流→异常流各生成用例 | 涉及多步骤业务流程 |
| **EG 错误推测** | `EG` | 补充特殊字符、极端值、并发场景 | 高风险模块、历史缺陷多 |
| **RV 出参验证** | `RV` | 逐字段验证响应体结构完整性 | 所有接口的正向用例 |
| **BS 业务状态** | `BS` | 对关键实体的每个状态枚举值生成独立用例 | 接口行为依赖实体状态 |
| **RL 关联关系** | `RL` | 验证凭证/资源的归属匹配与不匹配行为 | 接口涉及凭证与资源绑定 |
| **DB 数据库验证** | `DB` | 验证表结构 + 写操作后数据库记录正确性 | 新建表/新增字段/写操作接口 |
| **REDIS 缓存验证** | `REDIS` | 验证缓存生成/值/TTL/更新策略/删除行为 | 需求涉及缓存操作 |

## 需求追溯与覆盖率

> 详细规范见 [TRACEABILITY.md](references/TRACEABILITY.md)

- **需求ID**：自动识别 `REQ-xxx`、`F1.2`、`US_042`、`PROJ-123` 等模式；无ID时生成 `MOD_{缩写}_{序号}`
- **双向追溯**：需求→用例 + 用例→需求，Phase 2.8 输出覆盖率统计
- **覆盖率目标**：≥ 95%，未覆盖需求在预览中高亮警告

## 优先级与回归分类

> 详细规则见 [TEST-PRIORITY.md](references/TEST-PRIORITY.md)

| 级别 | 来源 | 回归集 |
|------|------|--------|
| **P0** 核心 | 基本流用例 | 冒烟测试（每次构建） |
| **P1** 主要 | 备选流 + 边界值 | 核心回归（每日/提测） |
| **P2** 次要 | 异常流 + 错误推测 | 全量回归（发版前） |
| **P3** 边缘 | 边缘错误推测 | 全量回归（发版前） |

## .memory 记忆系统

> 完整 Schema 定义见 [MEMORY-SCHEMA.md](references/MEMORY-SCHEMA.md)

Skill 在项目中创建 `.memory/` 文件夹，存储跨会话学习数据：

```
.memory/
├── project-context.json      # 项目上下文（路径、名称）
├── terminology.json          # 领域术语库（自动学习 + 手动补充）
├── generation-history.json   # 生成历史（质量趋势分析）
├── user-preferences.json     # 用户偏好（交互模式、步骤粒度等）
└── ambiguity-decisions.json  # 歧义决策记录（避免重复询问）
```

**核心机制**：
- 每次生成自动写入 `generation-history.json`（含覆盖率、优先级分布、用例数）
- 歧义处理决策写入 `ambiguity-decisions.json`，后续相似歧义自动复用
- 用户修正的术语、偏好自动持久化，下次会话即刻生效

## Workflow

### Phase 0: 项目初始化（首次运行）
1. 检测项目结构（扫描常见需求目录和输出目录）
2. 创建 `.memory/` 文件夹（`scripts/memory_manager.py --action init`）
3. 从需求文档提取领域术语，存入 `terminology.json`
4. 保存用户偏好到 `user-preferences.json`

### Phase 1: 读取需求

**需求目录确认**：需求文档的存放位置由用户告知，不硬编码固定路径。确认逻辑如下：
1. 检查 `project-context.json` 中是否已存储 `requirements_dir`（上次使用的需求目录）
2. 若已存储且目录存在 → 直接使用，在摘要中提示 `ℹ️ 需求目录：{requirementsDir}`
3. 若未存储或目录不存在 → 询问用户：`请提供需求文档所在目录路径（如 requirements/、docs/req/ 等）：`
4. 将用户确认的目录路径存入 `project-context.json` 的 `requirements_dir` 字段，后续会话自动复用

> 后文中 `{requirementsDir}` 均指用户确认的需求文档目录。

**输出目录确认**：用例文件和质量报告的输出位置由用户告知。确认逻辑如下：
1. 检查 `project-context.json` 中是否已存储 `output_dir`
2. 若已存储且目录存在 → 直接使用
3. 若未存储或目录不存在 → 询问用户：`请提供用例文件输出目录路径（如 test-docs/、output/ 等）：`
4. 将用户确认的目录路径存入 `project-context.json` 的 `output_dir` 字段

> 后文中 `{outputDir}` 均指用户确认的输出目录。

**支持的文件格式**：

| 类别 | 格式 | 读取方式 |
|------|------|---------|
| 文本文件 | `.md`、`.txt`、`.rst`、`.html`、`.htm` | 直接读取文件内容 |
| Office 文档 | `.docx`、`.xlsx`、`.pptx` | 通过 Python 脚本提取文本（`scripts/extract_text.py`） |
| PDF 文档 | `.pdf` | 通过 Python 脚本提取文本（`scripts/extract_text.py`） |
| 图片文件 | `.png`、`.jpg`、`.jpeg`、`.gif`、`.webp`、`.bmp`、`.svg` | 作为多模态输入 |

**读取步骤**：
1. 扫描 `{requirementsDir}` 目录下所有文件，按文件名排序
2. **文本文件**（`.md`、`.txt`、`.rst`、`.html`、`.htm`）：直接读取内容，文件间以 `---` 分隔，拼接为需求文本
3. **Office / PDF 文档**（`.docx`、`.xlsx`、`.pptx`、`.pdf`）：调用 `scripts/extract_text.py` 提取文本内容，追加到需求文本中
   - 调用方式：`python3 "${SKILL_ROOT}/scripts/extract_text.py" --input "{文件路径}" --output "{临时txt路径}"`
   - 提取后的文本与其他需求文本一起拼接
   - **降级策略**：若提取失败（缺少依赖库、文件损坏等），跳过该文件，并在 Phase 2.5 摘要中提示 `⚠️ 文档 {文件名} 文本提取失败，已跳过`
4. **图片文件**（`.png`、`.jpg`、`.jpeg`、`.gif`、`.webp`、`.bmp`、`.svg`）：与需求文本一起作为多模态输入
5. **降级策略**：若某个文件解析失败（格式不支持、文件损坏等），跳过该文件，基于其他文件内容生成用例，并在 Phase 2.5 摘要中提示 `⚠️  {文件名} 解析失败，已跳过`
6. 以上述内容进入 Phase 2 解析


### Phase 2: 解析需求
1. **加载记忆并应用**：
   - 读取 `project-context.json` → 获取项目名称、需求目录、输出目录等上下文
   - 读取 `terminology.json` → 解析时统一术语，避免同义词重复
   - 读取 `ambiguity-decisions.json` → 相似歧义自动复用历史决策，跳过询问
   - 读取 `generation-history.json` → 提取历史优先级分布作为基线，识别历史常见遗漏场景类型并主动补充
   - 读取 `user-preferences.json` → 应用用户步骤粒度、标题风格等偏好
2. **提取需求ID**：识别或生成需求唯一标识
3. **结构识别**：功能模块、验收条件、业务规则
4. **系统性测试设计**（按 EP→BVA→PV→ST→EG 顺序叠加核心方法，再按 RV→BS→RL→DB→REDIS 补充深度覆盖）：
   - 等价类划分：识别所有输入的有效/无效等价类
   - 边界值提取：识别数值、长度、时间边界
   - 场景流识别：基本流、备选流、异常流
   - 错误推测补充：根据输入类型和模块风险等级触发
   - 业务规则补充：参考 [BUSINESS-RULES.md](references/BUSINESS-RULES.md)，匹配需求涉及的功能类型（列表、输入框、按钮等）自动追加对应规则用例
   - 公司特定规则：参考 [COMPANY-NOTES.md](references/COMPANY-NOTES.md)，排除网关注入参数等公司环境特有的约束
5. **去重扫描**：不同方法产生的用例可能语义重复（如 EP 无效类与 EG 错误推测场景重叠），合并等价用例，保留更具体的那条。
   - **状态变体去重**：若某条规则已被一个清晰的用例验证，不要再以"不同数据状态"为由生成同等验证目标的变体用例。
     - ✅ 已有「系统报表展示【系统】标签」+ 「自定义报表不展示【系统】标签」→ **不需要再生成**「仅存在自定义报表时不显示标签」「新增系统报表后自动显示标签」「删除系统报表后自定义报表仍不显示标签」，因为验证目标完全相同，只是前置数据状态不同
     - 判断标准：**新用例的验证目标（"验什么"）是否已被现有用例覆盖**，若是则丢弃；只有"验什么"不同才保留
   - **场景修饰词变体去重**：不要在已有用例的基础上，通过添加"混合时""同时存在时""共存时"等场景修饰词生成新用例，若验证目标实质相同则合并。
     - ✅ 已有「系统汇总表排在上方自定义汇总表排在下方」→ **不需要再生成**「存在系统表和自定义表混合时排序规则正确」，因为"混合场景"本就是前者的前置条件，两者验证目标完全相同
     - 判断标准：去掉修饰词后，两条用例核心断言（"系统在上、自定义在下"）是否一致，一致则合并，保留表述更明确的那条
6. **歧义检测**：识别不明确的需求描述；与 `ambiguity-decisions.json` 比对，相似歧义自动复用历史决策
7. **覆盖率预计算**：统计需求→用例映射，标记未覆盖需求
8. 参考 [PARSING-RULES.md](references/PARSING-RULES.md)

### Phase 2.5: 【检查点1】解析确认
使用 **AskUserQuestion** 确认解析结果：

```markdown
已识别 X 个模块，Y 条需求，Z 条规则

仅在发现警告时询问用户
无警告则自动继续
```

### Phase 2.6: 歧义处理
对检测到的歧义需求逐一询问：

```markdown
需求原文：{text}
歧义类型：边界不明确 / 规则冲突 / 条件缺失
我的理解：{interpretation}

请选择：
1. 接受我的理解
2. 提供不同解释
3. 跳过此需求
4. 标记为待确认
```

- 仅询问影响 P0/P1 的关键歧义，其他歧义自动采用 AI 理解并标注

### Phase 2.8: 【检查点2】质量预审 & 生成预览

本阶段分为 3 个**必须按顺序执行**的子步骤。不允许跳过任何子步骤。

#### Step 2.8.1: 质量自检（内部执行，不输出给用户）

对已设计的用例集执行以下检查清单，不通过则自动修正后重新检查，最多重试 3 次：

```
□ 覆盖率 ≥ 95%（未覆盖需求列表高亮）
□ 参数验证覆盖率：每个输入参数至少覆盖必填性和数据类型验证
□ P0 占比 10-15%（用例总数 < 10 时不强制，改为建议性提示）
□ P1 占比 30-40%（用例总数 < 10 时不强制，改为建议性提示）
□ 每条需求至少关联 1 种设计方法
□ 设计方法叠加完整性：每条需求已按 EP→BVA→PV→ST→EG→RV→BS→RL→DB→REDIS 顺序逐一评估，符合触发条件的方法均已应用
□ 无需求外编造的场景
□ 无语义重复用例
□ 出参验证覆盖率：每个接口至少 1 条 RV 用例（接口文档驱动时）
□ 业务状态覆盖率：每个状态枚举的每个值至少 1 条 BS 用例
□ 关联关系覆盖率：每对凭证-资源关联至少 1 条匹配 + 1 条不匹配用例
□ 数据库验证覆盖率：每个写操作接口至少 1 条 DB-DML 用例；新建表每个字段 1 条 DB-DDL 用例
□ Redis缓存验证覆盖率：每个涉及缓存操作的接口至少 1 条 REDIS 用例；每个缓存 key 至少覆盖缓存生成+缓存值+缓存时间 3 个维度
□ 负面用例占比 ≥ 30%，每个功能模块至少 1 条异常流用例
□ 功能覆盖：需求文档中每条需求项/验收条件都至少被 1 条用例覆盖
□ 边界覆盖：每个有边界的输入参数都包含有效/无效边界值用例
□ 异常场景覆盖：每个功能模块都覆盖了错误输入、异常操作场景
□ 需求分支覆盖率 ≥ 90%（基于需求中的条件分支数统计，如 if/else、多状态、多角色等分支）
□ 模块均衡性检查（详见下方规则）
```

**模块均衡性检查 (Module Balance Check)**：

> **优先级说明**：当模块均衡性检查与整体优先级分布指标冲突时（如补充 P0 冒烟用例导致 P0 占比超过 15%），以整体分布指标为准。此时零用例模块的冒烟用例可降为 P1。

| 检查项 | 规则 | 修正动作 |
|--------|------|----------|
| **零用例模块** | 若识别出 N 个功能模块，某模块用例数为 0 | 强制阻断，要求针对该模块至少生成 1 条冒烟用例 (P0)；若补充后 P0 占比超过 15%，将多余的 P0 降为 P1 |
| **PV 缺失** | 若存在输入参数但 PV 设计方法用例数为 0 | 强制补充，为每个输入项生成至少 1 条 PV 用例 |
| **方法单一** | 若某模块仅使用了 1 种设计方法（如只有 ST） | 强制补充，至少补充 1 种方法（如 BVA 或 EP） |

**设计方法叠加完整性检查 (Method Coverage Check)**：

对每条需求，逐一检查 10 种设计方法是否已按触发条件评估并应用：

| 方法 | 触发条件 | 检查规则 |
|------|---------|---------|
| EP | 有输入范围、格式、枚举约束 | 若需求含输入约束但无 EP 用例 → 强制补充 |
| BVA | 有数值/长度/时间边界 | 若需求含边界值但无 BVA 用例 → 强制补充 |
| PV | 有输入参数的接口 | 若接口有入参但无 PV 用例 → 强制补充 |
| ST | 涉及多步骤业务流程 | 若需求含业务流程但无 ST 用例 → 强制补充 |
| EG | 高风险模块、历史缺陷多（默认高风险：涉及金额计算、权限校验、数据删除的模块；首次使用无历史数据时按此默认规则判断） | 若模块为高风险但无 EG 用例 → 建议补充 |
| RV | 所有接口的正向用例 | 若接口有响应体定义但无 RV 用例 → 强制补充 |
| BS | 接口行为依赖实体状态 | 若需求含状态枚举但无 BS 用例 → 强制补充 |
| RL | 接口涉及凭证与资源绑定 | 若接口含凭证+资源但无 RL 用例 → 强制补充 |
| DB | 新建表/新增字段/写操作接口 | 若接口含写操作但无 DB 用例 → 强制补充 |
| REDIS | 需求涉及缓存操作（显式提及"缓存""Redis""TTL"，或设计文档中有 Redis 设计表格；需求提到"实时性""高性能查询""频繁读取"时建议评估是否需要 REDIS 用例） | 若需求提及缓存但无 REDIS 用例 → 强制补充 |

> 不符合触发条件的方法无需生成用例，但必须经过评估（即"看过了，不适用"也算通过）。

**补充后去重**：Phase 2.8 补充用例后，必须执行一次轻量去重检查（仅对新补充的用例与已有用例做语义等价判断），避免补充引入新的重复。

**熔断机制**：质量自检最多重试 3 次。若 3 次修正后仍有指标未达标，进入 Step 2.8.2 时在报告中标注未达标项。

### Phase 3: 生成文档
1. **分配优先级**：根据用例类型自动判断 P0-P3
2. **关联需求ID**：每个用例记录来源需求
3. **用例写作规范**：
   - 标题格式：`动作 + 对象 + 条件/场景`，禁用模糊词（“正常”“正确”）
   - 步骤粒度：每步只做一件事，禁止多动作合并（如"输入用户名并点击登录"应拆为两步）；步骤只写操作，禁止在步骤中使用"验证""检查""确认"等验证类动词
   - 步骤编号：每条操作以 `N. ` 起始（如 `1. 打开登录页面`），对应预期结果使用**相同编号**（如 `1. 显示账号密码输入框`），编号从 1 开始连续递增，steps 与预期结果数组长度必须相等
   - 预期结果：必须可验证，包含具体数据/状态，禁用"系统正常处理""正常显示"等模糊描述；一个步骤有多个验证点时用逗号连接写在同一条预期结果中，不拆分步骤
   - 前置条件：仅当需要特定环境准备时填写，避免写"用户已登录"这类显而易见的内容
   - 业务规则扩充（[BUSINESS-RULES.md](references/BUSINESS-RULES.md)）属于合理推断范围，不受"无编造"约束限制。判定标准：补充的规则必须是该功能类型的行业通用行为（如"删除前二次确认""列表支持分页"），且不引入需求中未提及的新功能点（如需求只写"导出"，不得自行补充"支持 Excel/PDF/CSV 三种格式"）
   - 应用 `user-preferences.json` 中存储的步骤粒度、标题风格偏好
4. 将用例组织为 JSON 数组，写入项目临时文件 `tmp/cases_<时间戳>.json`（如 `tmp/cases_20260225143022.json`），目录不存在时自动创建
  - 时间戳必须为 **14 位**：`YYYYmmddHHMMSS`（正则：`^\d{14}$`）
  - 禁止仅日期命名（如 `tmp/cases_20260226.json`）
4. 调用 `scripts/generate_xmind.py -f tmp/cases_<时间戳>.json` 生成 XMind 文件，输出路径为 `{outputDir}/testcases_<时间戳>.xmind`（`{outputDir}` 为用户指定的输出目录）

### Phase 4: 更新记忆（自学习闭环）

> 完整学习规则见 [LEARNING-RULES.md](references/LEARNING-RULES.md)

1. **生成历史**：写入 `generation-history.json`，包含覆盖率、优先级分布、用例数、来源文件
2. **歧义决策**：将 Phase 2.6 中用户的歧义处理决策写入 `ambiguity-decisions.json`，后续相似歧义自动复用
3. **新术语**：如解析中发现新术语，更新 `terminology.json`
4. **用户偏好**：保存用户偏好到 `user-preferences.json`
5. **质量趋势**：与历史记录对比，生成趋势摘要（如覆盖率是否提升、用例数变化）

### Phase 5: 用户反馈学习（生成后可选）

用户对生成结果提出修改或报告测试中发现的 bug 时，自动触发学习：

#### 5.1 用例修正类反馈

| 反馈类型 | 触发示例 | 学习动作 |
|----------|---------|----------|
| 纠正逻辑错误 | "这条用例的预期结果不对" | 重新生成该用例 + 记录歧义模式到 `ambiguity-decisions.json` |

| 修改术语 | "这里应该叫 XXX" | 询问"是否记住？"→ 写入 `terminology.json` |

| 调整模块分组 | "这几条应该归到同一个模块" | 调整分组 + 询问"是否记住这个分组规则？" |

| 补充预期结果 | "预期结果不够具体" | 补充具体预期 + 记录为写作风格偏好 |

#### 5.2 Bug 反向学习（核心闭环）

当用户报告测试中发现了新 bug，且该 bug 未被已生成的用例覆盖时，触发 bug 反向学习流程。

**触发词**：用户说类似"为什么没有生成 xxx 的用例""为什么遗漏了 xx bug""测试发现了一个 bug，用例没覆盖到""这个 bug 为什么没测到"

**学习流程**：

```
用户报告 bug / 遗漏
    |
Step 1: 分析 bug 根因
  - 这个 bug 属于哪个功能模块？
  - 触发条件是什么？（输入、状态、时序、边界等）
  - 预期行为 vs 实际行为是什么？
    |
Step 2: 回溯遗漏原因（输出给用户）
  从 10 种设计方法的角度逐一分析为什么遗漏：
  - EP：是否遗漏了某个等价类？（如未考虑某种输入类型）
  - BVA：是否遗漏了某个边界值？（如未测试 0 值、负数）
  - PV：是否遗漏了某个参数验证？（如未验证字段组合）
  - ST：是否遗漏了某条业务流程路径？（如未覆盖某个分支）
  - EG：是否属于经验性场景？（如并发、重复提交）
  - RV：是否遗漏了某个响应字段验证？
  - BS：是否遗漏了某个业务状态？
  - RL：是否遗漏了某个关联关系校验？
  - DB：是否遗漏了数据库层面的验证？
  - REDIS：是否遗漏了缓存层面的验证？
    |
Step 3: 生成补充用例
  - 针对该 bug 生成 1-3 条补充用例
  - 标记设计方法和需求ID
  - 追加到已有用例集中
    |
Step 4: 提出改进建议（输出给用户）
  - 分析遗漏的根本原因（需求描述不清？设计方法触发条件不够？业务规则未覆盖？）
  - 给出改进 skill 的具体建议（如"建议在 BUSINESS-RULES.md 中补充 xxx 规则"）
  - 询问用户是否同意执行改进
    |
Step 5: 持久化学习
  - 将 bug 模式记录到 generation-history.json（含 bug 描述、遗漏原因、补充用例）
  - 按频次规则判断是否升级为"常见遗漏"
  - 若用户同意改进建议 -> 更新对应的 references 文档（如 BUSINESS-RULES.md、PARSING-RULES.md 等）
```

**Bug 记录格式**（写入 `generation-history.json`）：
```json
{
  "type": "bug_learning",
  "date": "ISO datetime",
  "bug_description": "当数量为0时，接口返回500而非业务错误码",
  "module": "订单模块",
  "miss_reason": "BVA 未覆盖 0 值边界",
  "design_method_gap": "BVA",
  "supplemented_cases": ["BVA-边界值: 数量为0时返回参数错误"],
  "improvement": "BUSINESS-RULES.md 补充数量字段必须验证0值"
}
```

```
用户反馈 ───→ 分发到以上各文件 ──────────→ 下次生成自动避免相同问题
```

**记忆命令**：`"更新术语表"` / `"清除记忆"` / `"查看偏好"` → 调用 `scripts/memory_manager.py` 对应 action。

## 约束

### 质量底线
- **需求追溯**：所有用例必须关联需求ID，不允许悬空用例
- **方法标记**：所有用例必须标记至少 1 种设计方法
- **优先级分布**：P0: 10-15%, P1: 30-40%，P2 补足剩余，P3 ≤ 20%（异常时 Phase 2.8 拦截）
- **覆盖率**：≥ 95%（未达标时 Phase 2.8 高亮警告）
- **无编造**：不凭空编造需求中不存在的场景
- **无冗余**：不同设计方法产生的语义重复用例必须合并

### 交互规则
- 仅在发现问题/异常时询问用户
- 无异常时自动继续执行

### 其他
- 遵循项目已有命名规范
- 默认输出中文，除非项目配置为其他语言
- `.memory` 文件夹应加入 .gitignore
- **禁止复用 tmp/ 缓存**：每次生成均从 Phase 1 重新开始，不得读取 `tmp/` 中已有的 JSON 文件跳过生成步骤
