# 学习规则

定义 Skill 如何学习和改进的规则。

## 用户偏好学习

### 触发条件

偏好学习仅在以下**明确信号**出现时触发，避免误触发：

| 触发信号 | 置信度 | 处理方式 |
|---------|--------|---------|
| 用户明确说"以后都按这个来""记住这个偏好""默认用这个" | 高 | 直接写入，无需询问 |
| 用户连续 2 次以上手动修改同一类内容（如标题格式、步骤粒度） | 高 | 直接写入，无需询问 |
| 用户单次修改生成内容（如调整步骤粒度、修改标题风格） | 低 | 暂存，对话末尾批量确认 |

> **不触发**：用户的一般性评价（如"挺好的""还行"）不视为偏好表达，不触发学习流程。

### 学习流程

```
用户修改生成内容 / 表达偏好
    ↓
判断置信度
    ↓
高置信度 → 直接写入 user-preferences.json
低置信度 → 暂存到待确认列表
    ↓
对话末尾（或生成完成后）批量确认：
  "本次有以下偏好变化，是否记住？"
  - [1] 步骤粒度：合并为粗粒度
  - [2] 标题风格：动作+对象+条件
  请输入要记住的编号（如 1,2）或输入 0 跳过全部
    ↓
用户选择 → 写入 / 跳过
```

### 可学习的偏好

| 类型 | 示例 |
|------|------|
| 用例标题风格 | "输入X，结果Y" 格式 |
| 步骤粒度 | 操作步骤合并 or 拆分 |
| 模块层级深度 | 最多 3 层 / 最多 5 层 |
| 预期结果风格 | 每步一条 / 汇总写 |

## 术语学习

### 触发条件
- 解析需求文档时发现专业术语
- 用户在对话中解释术语或修正叫法

### 识别模式

| 模式 | 示例 | 置信度 |
|------|------|--------|
| 括号解释 | `GMV（商品交易总额）` | 高 → 直接写入 |
| 冒号解释 | `SKU：库存单位` | 高 → 直接写入 |
| 引号定义 | `"SKU" 指库存单位` | 高 → 直接写入 |
| 首字母大写缩写 | `PV`、`UV` | 低 → 标记为待确认 |

> **待确认术语处理**：自动检测到的首字母大写缩写不直接写入 `terminology.json`，而是在 Phase 2.5 摘要中列出待确认术语，由用户确认后再写入。常见通用缩写（如 OK、APP、ID、API、URL、HTTP）自动排除，不作为术语候选。

### 学习内容
- **领域术语**：专业名词及含义、缩写全称
- **模块缩写**：模块名称 → 缩写映射，用于生成用例编号

### 存储格式（terminology.json 示例）
```json
{
  "domain_terms": {
    "GMV": "商品交易总额",
    "SKU": "库存单位"
  },
  "module_abbreviations": {
    "登录": "LOGIN",
    "订单": "ORDER"
  }
}
```

## 用户反馈学习

### 反馈类型

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 纠正错误 | "这条用例逻辑不对" | 重新生成 + 记录歧义模式到 `ambiguity-decisions.json` |
| 删减用例 | "P2 太多了" | 询问"是否调整优先级分布作为默认？"→ 写入 `user-preferences.json` |
| 补充场景 | "还缺并发场景" | 增补用例 + 记录到 `generation-history.json`（见下方频次规则） |
| 修改术语 | "这里应该叫 XXX" | 询问"是否记住？"→ 写入 `terminology.json` |
| 调整步骤 | "步骤太细了，合并一下" | 按置信度处理（见偏好学习流程） |

### 常见遗漏的频次规则

单次用户补充不等于"常见遗漏"。遗漏场景的升级规则：

| 补充次数 | 状态 | 后续行为 |
|---------|------|---------|
| 第 1 次 | 记录为"单次补充" | 仅记录，不影响后续生成 |
| 第 2 次（同类场景） | 记录为"重复补充" | 仅记录，不影响后续生成 |
| 第 3 次及以上（同类场景） | 升级为"常见遗漏" | 后续生成同类模块时主动补充该场景 |

> "同类场景"的判断标准：场景类型相同（如"并发场景""超时场景"），不要求需求文本完全一致。

### 反馈触发时机

生成完成后，用户对结果的任何修正建议都应触发学习流程（Phase 5）。AI 必须：
1. 先执行用户的修正请求
2. 低置信度偏好暂存，对话末尾批量确认
3. 高置信度偏好直接写入对应 `.memory/` 文件

## 持续改进

### 生成质量
1. **分析历史记录**：统计常见模块、识别重复遗漏的场景类型（需达到频次阈值）
2. **优化场景覆盖**：确保基本流 / 备选流 / 异常流 / 边界值均有覆盖
3. **复用模块结构**：相似模块共享测试模式，减少重复设计

### 效率
1. **缓存常用术语**：跨会话复用 `terminology.json`，避免重复提取
2. **应用用户偏好**：基于 `user-preferences.json` 自动应用已确认的偏好，减少重复确认

## 记忆管理命令

| 用户指令 | 对应脚本操作 | 说明 |
|---------|------------|------|
| `更新术语表` | `scripts/memory_manager.py update --type terminology` | 重新扫描术语 |
| `清除记忆` | `scripts/memory_manager.py clear` | 重置所有学习数据 |
| `重置偏好` | `scripts/memory_manager.py clear --type preferences` | 仅重置用户偏好 |
| `重置术语` | `scripts/memory_manager.py clear --type terminology` | 仅重置术语库 |
| `重置歧义决策` | `scripts/memory_manager.py clear --type ambiguity` | 仅重置歧义决策记录 |
| `重置生成历史` | `scripts/memory_manager.py clear --type history` | 仅重置生成历史 |
| `显示学习记录` | `scripts/memory_manager.py read` | 查看历史生成与偏好 |
| `查找歧义` | `scripts/memory_manager.py find-ambiguity --keyword <词>` | 查询已记录的歧义处理 |
