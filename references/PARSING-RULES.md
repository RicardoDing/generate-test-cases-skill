# 需求解析规则

从需求文档中提取测试场景的规则和模式。各设计方法的详细定义参见 [TEST-DESIGN-METHODS.md](TEST-DESIGN-METHODS.md)。

## 需求文档结构识别

### 常见需求文档格式

**格式1：PRD 结构**
```
# 产品需求文档
## 1. 背景
## 2. 功能列表
### 2.1 功能A
### 2.2 功能B
## 3. 非功能需求
```

**格式2：用户故事**
```
作为 <角色>
我想要 <功能>
以便于 <价值>

验收条件：
- AC1
- AC2
```

**格式3：功能说明**
```
## 功能名称
### 功能描述
### 输入
### 输出
### 业务规则
### 异常处理
```

## 关键信息提取

### 1. 功能模块
- 识别一级/二级标题
- 提取模块名称和层级关系

### 2. 功能描述
- 找到功能说明段落
- 提取核心行为描述

### 3. 验收条件
关键词识别：
- "验收条件"、"AC"、"Acceptance Criteria"
- "应该"、"必须"、"需要"
- 编号列表（1. 2. 3. 或 - ）

### 4. 业务规则
关键词识别：
- "规则"、"限制"、"约束"
- "如果...则..."、"当...时..."
- 数值限制（"最多"、"至少"、"范围"）

### 5. 异常场景
关键词识别：
- "异常"、"错误"、"失败"
- "如果不"、"当...无效时"
- "超时"、"中断"、"取消"

### 6. 边界值
识别模式：
- 数值范围（"1-100"、"最大100"）
- 长度限制（"最多50字符"）
- 时间限制（"30分钟内"）
- 数量限制（"最多5次"）

## 测试场景生成规则

### 从验收条件生成
```
验收条件 → 正向测试用例（验证条件满足时的行为）
验收条件取反 → 反向测试用例（验证条件不满足时的处理）
```

### 从边界值生成

对于范围 `[MIN, MAX]`（如"数量 1-100"），分别对上下边界生成测试点：

```
下边界 MIN：
- MIN-1（边界外，无效值）
- MIN（边界上，有效值）
- MIN+1（边界内，有效值）

上边界 MAX：
- MAX-1（边界内，有效值）
- MAX（边界上，有效值）
- MAX+1（边界外，无效值）

特殊边界：
- 0 或空值
```

> 关键区别：N-1 和 N+1 的有效性取决于是上边界还是下边界，不能一概而论。


### 从业务规则生成
```
规则条件 A：
- 满足 A 时的行为
- 不满足 A 时的行为
- A 的边界情况
```

## 示例

**需求原文**：
> 用户注册时需要输入手机号，手机号必须是11位数字且以1开头。
> 如果手机号已被注册，提示"该手机号已注册"。
> 验证码60秒内只能发送一次。

**提取结果**：

| 提取项 | 内容 |
|-------|------|
| 功能 | 用户注册 |
| 输入 | 手机号 |
| 规则1 | 11位数字 |
| 规则2 | 以1开头 |
| 异常1 | 手机号已注册 |
| 限制 | 60秒发送间隔 |

**生成的测试场景**：

| 编号 | 场景 | 类型 | 来源 |
|-----|------|------|------|
| 1 | 输入有效手机号注册成功 | 正向 | 规则1+2 |
| 2 | 输入10位手机号 | 反向 | 规则1边界 |
| 3 | 输入12位手机号 | 反向 | 规则1边界 |
| 4 | 输入非1开头的号码 | 反向 | 规则2 |
| 5 | 输入已注册手机号 | 异常 | 异常1 |
| 6 | 60秒内重复发送验证码 | 边界 | 限制 |
| 7 | 60秒后再次发送验证码 | 边界 | 限制 |

## 场景覆盖检查

生成用例后需检查：
- [ ] 每个验收条件都有对应的正向用例
- [ ] 每个验收条件都有对应的反向用例
- [ ] 每个边界值都有边界测试
- [ ] 每个异常场景都有对应用例
- [ ] 用例可追溯到需求来源
- [ ] 无语义重复用例

## 用例去重规则

4 种设计方法叠加后可能产生语义重复用例，需在生成前去重：

### 常见重复模式

| 重复模式 | 示例 | 处理方式 |
|---------|------|---------|
| EP 无效类 ≈ EG 错误推测 | EP"输入非数字字符" ≈ EG"输入特殊字符" | 合并保留 EG（更具体） |
| BVA 离点 ≈ EP 无效类边界 | BVA"长度5位" ≈ EP"长度<6位" | 合并保留 BVA（有明确值） |
| ST 异常流 ≈ EG 错误推测 | ST"支付超时" ≈ EG"网络超时" | 合并保留 ST（有完整流程） |

### 去重原则

1. **语义等价判断**：两条用例的`操作`和`预期结果`本质相同，仅措辞不同 → 合并
2. **保留更具体的**：合并时优先保留有具体测试数据/完整步骤的用例
3. **完全等价兜底**：当两条用例完全等价、无法判断谁更具体时，按以下设计方法优先级保留排名靠前的：
   `ST > BVA > EP > PV > EG > RV > BS > RL > DB > REDIS`
   （场景法有完整流程 > 边界值有明确数据 > 等价类有分类依据 > 参数验证有规则 > 其余按补充方法顺序）
4. **合并设计方法标签**：合并后的用例标记所有来源方法，如 `["EP", "EG"]`
5. **不跨需求合并**：不同需求ID产生的类似用例保持独立

## 设计方法应用（识别触发）

> 各方法的理论定义、用例生成原则和标准示例详见 [TEST-DESIGN-METHODS.md](TEST-DESIGN-METHODS.md)。
> 本节仅描述**从需求文本触发各方法的识别模式**。

### 等价类划分 — 触发关键词

| 模式类型 | 关键词/句式 | 划分方向 |
|---------|------------|---------|
| 范围限制 | `在...之间`、`N-M`、`不超过`、`至少` | 有效区间 / 超出区间 |
| 集合限制 | `仅限`、`只能是`、`可选值` | 集合内 / 集合外 |
| 格式限制 | `格式`、`正则`、`必须为` | 符合 / 不符合 |
| 必填标记 | `必填`、`不能为空` | 有值 / 空值 |
| 长度限制 | `位数`、`字符数`、`最短`、`最长` | 有效长度 / 太短 / 太长 |
| 类型限制 | `数字`、`字母`、`整数`、`小数` | 正确类型 / 错误类型 |

### 边界值分析 — 触发关键词

| 边界类型 | 关键词 | 提取方法 |
|---------|--------|---------|
| 数值边界 | `最小`、`最大`、`≤`、`≥`、`<`、`>` | N-1, N, N+1 |
| 长度边界 | `长度`、`字符数`、`位数` | M-1, M, N, N+1 |
| 数量边界 | `最多N个`、`至少M条`、`限制N次` | M-1, M, N, N+1 |
| 时间边界 | `N分钟内`、`超过N小时` | N-1, N, N+1 |
| 金额边界 | `最低N元`、`上限M` | N-0.01, N, M, M+0.01 |

### 场景法 — 流程识别模式

| 流程类型 | 识别模式 |
|---------|---------|
| 基本流 | 无条件分支的陈述句、`成功后` |
| 备选流 | `也可以`、`或者`、`另外支持`、`如果选择` |
| 异常流 | `如果失败`、`当...无效时`、`超时`、`否则` |
| 条件分支 | `如果<条件>则<A>，否则<B>` → 两条路径 |
| 多方式 | `支持 A / B / C` → 各一条备选流 |

### 错误推测 — 自动触发条件

| 需求元素 | 自动补充的推测用例 |
|---------|------------------|
| 输入框 | SQL注入、XSS、特殊字符、超长输入、空值 |
| 提交按钮 | 重复提交、快速双击、并发操作 |
| 文件上传 | 非法格式、超大文件、空文件 |
| 金额字段 | 小数精度、负数、0元 |
| 日期时间 | 跨天/跨年、无效日期、时区 |
| 列表/表格 | 首条、末条、空列表、大数据量 |

**模块级推测**（根据模块类型自动补充）：

| 模块 | 推测场景 |
|------|---------|
| 登录 | 账号不存在、会话过期、密码大小写 |
| 注册 | 重复注册、协议未勾选 |
| 支付 | 余额不足、重复支付、超时 |
| 订单 | 库存不足、并发下单 |
| 权限 | 越权访问、Token过期 |
| 搜索 | 空结果、特殊字符搜索、超长关键词 |
| 消息/通知 | 未读计数、消息已读状态、推送失败 |
| 购物车 | 商品下架、数量超库存、价格变动 |
| 导入/导出 | 空文件、格式错误、超大文件、编码问题 |
| 审批流 | 审批人不存在、重复审批、撤回后再审批 |

## 需求ID提取规则

### 自动识别模式

| 模式名称 | 正则表达式 | 示例匹配 |
|---------|-----------|---------|
| 标准REQ | `REQ[-_]?\d{3,}` | REQ_001, REQ-123 |
| 功能编号 | `F[-_]?\d+(\.\d+)*` | F1.2, F-3.1.2 |
| 用户故事 | `US[-_]?\d{3,}` | US_042, US-001 |
| 章节编号 | `^\d+(\.\d+)+\s+` | 3.2.1 用户登录 |
| JIRA格式 | `[A-Z]{2,}-\d+` | PROJ-123, BUG-456 |
| 中文编号 | `需求\d+` | 需求001 |

### 需求ID生成规则

如果需求文档中没有明确的需求ID，按以下规则自动生成：

```
格式：MOD_{模块缩写}_{序号}
序号规则：模块内从 001 递增（非全局递增）
冲突检查：生成前扫描文档中已有的需求ID，确保自动生成的 ID 不与已有 ID 重复

示例：
- 登录模块第1个需求：MOD_LOGIN_001
- 订单模块第3个需求：MOD_ORDER_003
- 用户管理模块：MOD_USER_001

模块缩写映射：
- 登录/注册 → LOGIN
- 用户/个人中心 → USER
- 订单 → ORDER
- 支付 → PAY
- 商品/产品 → PRODUCT
- 购物车 → CART
- 搜索 → SEARCH
- 消息/通知 → MSG
- 设置/配置 → CONFIG
```

## 综合解析流程

Claude 解析需求文档时应按以下顺序执行：

```
1. 结构识别
   - 识别文档格式（PRD/用户故事/功能说明）
   - 提取模块/功能层级

2. 需求ID提取
   - 自动识别或生成需求ID
   - 建立需求列表

3. 等价类划分
   - 扫描所有输入项
   - 识别范围/集合/格式/必填限制
   - 生成等价类表

4. 边界值提取
   - 扫描数值、长度、时间限制
   - 提取边界点

5. 场景流识别
   - 识别基本流（主路径）
   - 识别备选流（条件分支）
   - 识别异常流（错误处理）

6. 错误推测补充
   - 根据输入类型触发推测
   - 根据模块类型触发推测
   - 补充安全性测试点

7. 用例生成
   - 整合所有测试点
   - 分配优先级
   - 关联需求ID
   - 标记设计方法

8. 歧义检测
   - 扫描模糊词汇
   - 检测规则冲突
   - 识别缺失信息
   - 标记待确认项
```

# 歧义检测规则

当解析需求文档时，Claude 应主动识别以下歧义模式，并使用 AskUserQuestion 询问用户澄清。

## 歧义类型定义

| 类型代码 | 类型名称 | 影响级别 | 处理优先级 |
|---------|---------|---------|-----------|
| `BOUNDARY_UNCLEAR` | 边界不明确 | 高 | P0/P1 |
| `RULE_CONFLICT` | 规则冲突 | 高 | P0/P1 |
| `MISSING_ERROR` | 缺少错误处理 | 中 | P1/P2 |
| `VAGUE_CRITERIA` | 模糊验收标准 | 中 | P1/P2 |
| `INCOMPLETE_FLOW` | 流程不完整 | 中 | P1/P2 |
| `UNDEFINED_TERM` | 未定义术语 | 低 | P2/P3 |

## 检测模式速查

### BOUNDARY_UNCLEAR（边界不明确）

**触发关键词**：`适当`、`合理`、`一定`、`足够`、`若干`、`较多`、`较少`、`多个`、`几次`、`一段时间`

| 原文示例 | 询问方向 |
|---------|---------|
| "密码应有足够的复杂度" | 询问具体规则（长度、字符类型） |
| "系统应在合理时间内响应" | 询问具体秒数 |
| "用户可输入多个标签" | 询问最大数量 |

### RULE_CONFLICT（规则冲突）

**触发模式**：同一字段有互斥约束、前后文矛盾

| 规则A | 规则B | 冲突点 |
|-------|-------|--------|
| "密码必须包含特殊字符" | "密码只能包含字母和数字" | 特殊字符要求矛盾 |
| "订单金额自动四舍五入" | "订单金额精确到分" | 精度处理矛盾 |

### MISSING_ERROR（缺少错误处理）

**触发模式**：有操作但无失败说明、有条件但无"否则"、涉及外部系统但无超时处理

| 原文示例 | 缺失的处理 |
|---------|-----------|
| "系统调用支付接口完成扣款" | 支付失败/超时 |
| "点击保存按钮提交数据" | 网络异常/提交失败 |

### VAGUE_CRITERIA（模糊验收标准）

**触发关键词**：`正常显示`、`正确处理`、`友好提示`、`美观`、`流畅`、`等功能`

### INCOMPLETE_FLOW（流程不完整）

**触发模式**：有开始无结束、有条件无后续、步骤间有跳跃

### UNDEFINED_TERM（未定义术语）

**触发模式**：首次出现的专业词汇、缩写无解释 → 自动写入 terminology.json

## 歧义处理策略

以下类型的歧义必须询问用户确认：
- **BOUNDARY_UNCLEAR**（边界不明确）：所有实例均询问
- **RULE_CONFLICT**（规则冲突）：所有实例均询问
- **MISSING_ERROR**（缺少错误处理）：仅当涉及 P0/P1 级别功能（如支付、订单、认证等核心业务）时询问

其他类型（VAGUE_CRITERIA、INCOMPLETE_FLOW、UNDEFINED_TERM）自动采用 AI 理解并标注 ⚠️。

**自学习闭环**：
1. 每次歧义决策写入 `.memory/ambiguity-decisions.json`
2. 下次遇到相似歧义时自动沿用历史决策
